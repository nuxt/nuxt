---
title: '<ServerOnly>'
description: Render content only on the server and preserve it as static HTML without client hydration.
---

The `<ServerOnly>` component renders slot content on the server via [server components](/docs/4.x/directory-structure/app/components#server-components) (NuxtIsland). The initial SSR response includes the server-rendered HTML. On client-side navigation, content is fetched from the server and displayed. The rendered content is still included in the HTML sent to the browser and visible in the page source.

Use for content that must be fixed at render time: build timestamps, server-generated public content, or any markup that should not change after hydration.

::note
When `<ServerOnly>` is used as a server component (NuxtIsland), the slot can also run on the client during hydration or client-side navigation (slots are passed from the parent). So values like `{{ new Date() }}` may differ between SSR and what you see after hydration or after navigating to the page. The tests only assert the format and the `data-server-only` marker, not an exact match of SSR content.
::

::warning
Do not use `<ServerOnly>` for secrets or PII, since rendered content is visible in the page HTML.
::

## Example

```vue [app/pages/example.vue]
<template>
  <footer>
    <ServerOnly>
      <!-- Rendered once at build/request time, never updated on client -->
      Generated at {{ new Date().toISOString() }}
    </ServerOnly>
  </footer>
</template>
```

## Use Cases

- **Build timestamps**: Show when the page was generated without the date updating after hydration
- **Server-generated public content**: Display values computed on the server that should not re-run on the client (visible in page source)
- **Static server markup**: Any content that must remain exactly as rendered on the server

## Comparison with Server Components

`<ServerOnly>` is implemented as a [server component](/docs/4.x/directory-structure/app/components#server-components) and uses NuxtIsland under the hood. It accepts arbitrary slot content without defining a separate `.server.vue` file. Use it when you need a quick wrapper for server-only markup, similar to how `<ClientOnly>` wraps client-only content.
